# v0.4.0 design — diagnostics and test-framework usability

This document plans **v0.4.0** work: better **failure reporting** and **xUnit ergonomics**, **without** changing scheduling semantics (replay/random/Arrive–Hit–Release behavior stays as in v0.3.x unless a follow-up explicitly scopes scheduler changes).

**Scope of this pass:** design only. **No production code changes** were made to produce this file.

**Explicit non-goals (unchanged):**

- No `TaskScheduler` replacement  
- No automatic interception of every `await`  
- No binary / IL rewriting  
- No exhaustive model checking as the default product story  

Related positioning is documented in [README.md](../../README.md), [CHANGELOG.md](../../CHANGELOG.md), and [docs/roadmap.md](../roadmap.md).

---

## 1. Context — what exists today

### 1.1 Public types involved

| Type | Role |
|------|------|
| `BraidRunException` | Thrown when a braid run fails; carries **seed**, **iteration**, **trace** (`IReadOnlyList<string>`), **schedule** (`IReadOnlyList<BraidStep>`, empty when random-only), optional **inner** exception, and **`BraidSchedulerDiagnostics?`**. |
| `BraidSchedulerDiagnostics` | Structured snapshot: **has replay**, **last matched step**, **waiting** / **held** workers (`BraidProbeWaitDiagnostic`), **unused replay steps**. |
| `BraidProbeWaitDiagnostic` | `readonly record struct` — `(WorkerId, ProbeName)`. |
| `BraidStep` / `BraidSchedule` | Typed schedule; **`ToReplayText()`** / **`Parse`** for canonical text. |

Trace entries are **opaque strings** accumulated by the scheduler (e.g. `worker-1 forked`, `worker-1 hit ready`, `worker-1 released at ready`). There is no public strongly typed `BraidTraceEntry` today.

### 1.2 Current failure-report flow (end-to-end)

1. **Failure detected** inside `Braid.Internal.BraidScheduler` (script mismatch, timeout, fork fault, validation error, etc.) or propagated from `Braid.RunAsync` after `CreateException`.
2. **`CreateException(message, inner)`** (scheduler, under lock):  
   - Copies **`_trace`** → trace snapshot.  
   - Copies configured **`_schedule`** → schedule snapshot (empty list when no typed schedule).  
   - **`resolvedMessage = AppendReplayState(message)`** — prepends failure reason with **“Next replay step”** / **“Next replay operation”** lines when a typed schedule exists.  
   - **`BuildDiagnosticSnapshot()`** → `BraidSchedulerDiagnostics` (waiting/held/unused/last matched).  
   - Returns **`new BraidRunException(resolvedMessage, seed, iteration, trace, schedule, inner, diagnostics)`**.
3. **Consumer-facing text:** callers and tests almost always use **`exception.ToString()`**, which:  
   - Emits **Message** (already includes replay-position prefix from step 2 when applicable).  
   - Emits **Seed**, **Iteration**.  
   - If **schedule non-empty**: human-readable **Schedule** lines, then **Replay text** via `BraidSchedule.Replay(..).ToReplayText()`, or an **unavailable** line when text representation is impossible.  
   - **`AppendSchedulerDiagnosticsLines`** — last matched step, waiting, held, unused (or swallowed → **“Scheduler diagnostics unavailable.”**).  
   - **Trace** (numbered).  
   - **Inner exception** — single-line type + message (not a full stack trace dump).

**Random-only runs:** schedule snapshot is empty; **`ToString()`** does not invent replay text (consistent with README / CHANGELOG). Diagnostics may still carry useful waiting/held state when the scheduler captured them.

### 1.3 How tests exercise reporting today

- **`BraidFailureReportTests`**, **`BraidRuntimeBoundaryTests`**, **`BraidSchedulerBreakingTests`**, **`BraidTraceAndLifecycleBreakingTests`**, and others **`Assert.ThrowsAsync<BraidRunException>`** then **`Assert.Contains(..., exception.ToString(), ...)`**.  
- **`BraidRunExceptionToStringIsStable`** asserts **`ToString()`** idempotence for a constructed exception.  
- Example **`UserOperationLimiterTests`** normalizes line endings on **`ToString()`** before assertions.

## 2. Pain points — real xUnit / IDE usage

1. **`Exception.Message` vs full picture** — Test explorers and some CI viewers prioritize **`Message`**. Today the richest narrative is often **`ToString()`**, while **`Message`** may embed multi-line **`AppendReplayState`** prefix, which is easy to miss or truncate in UI.  
2. **Copy-paste friction** — Developers want one block to paste into issues or into `BraidSchedule.Parse(...)`. Replay text is inside **`ToString()`** but surrounded by other sections; **no single named API** returns “paste-ready replay snippet” or “paste-ready full diagnostic bundle.”  
3. **Structured data vs strings** — **`SchedulerDiagnostics`** exists but tooling and samples rarely use it; everything funnels through **`ToString()`**. Assertions on **`ToString`** are brittle (substring checks).  
4. **Trace is unstructured** — Parsing trace lines for tooling or custom reporters requires convention-dependent **`string.Split`** / **`Contains`**, not a stable contract.  
5. **xUnit integration** — There is no first-party helper for **`ITestOutputHelper`**, **`Assert.Fail` message enrichment**, or optional **files/output** attachment patterns (depending on runner). Each project reinvents `output.WriteLine(ex.ToString())`.  
6. **Inner exception depth** — **`ToString()`** summarizes only **immediate** inner type/message; deep **`AggregateException`** or nested wrappers may need clearer optional formatting for failures in large suites.  
7. **Parallelism / noise** — High-volume **`WriteLine`** in parallel tests can interleave; a small **correlation id** or **single formatted block** helper would reduce confusion (**candidate**, not committed).

---

## 3. Goals for v0.4.0 (diagnostics only)

- Keep **determinism** and **scheduler semantics** unchanged; v0.4.0 is **reporting and usability**.  
- Make **replay text** and **scheduler snapshot** easier to obtain **without** fragile **`ToString`** substring hacks.  
- Improve **xUnit** story with **optional** dependency (separate package vs.in-box extension methods — **TBD in implementation**).  
- Document **“random-only failure → build a replay schedule”** workflow explicitly (honest limits already in README).

## 4. Candidate APIs (**not committed**)

The following are **design sketches**. Naming, surface area, and package split must be reviewed before implementation. **Nothing here is a commitment to ship.**

### 4.1 On or adjacent to `BraidRunException`

| Candidate | Idea |
|-----------|------|
| `TryGetReplayText(out string text, out string? error)` | Returns canonical replay text when **`Schedule`** is representable; avoids try/catch around `ToReplayText` logic duplicated in **`ToString()`**. |
| `ToReportString()` or `FormatReport(BraidRunExceptionOptions?)` | Stable, documented layout separate from **`Exception.ToString()`** contract (or explicitly document **`ToString()`** as alias). |
| `IReadOnlyList<BraidTraceLine> TraceLines` | **Optional** strongly typed view over trace (**breaking or additive** — needs API review). |

### 4.2 Standalone report DTO (**optional**)

| Candidate | Idea |
|-----------|------|
| `BraidFailureReport` | Immutable snapshot: seed, iteration, message, schedule, replay text?, diagnostics, trace lines, inner summary — built from **`BraidRunException`** for testing and serialization. |
| `BraidSchedulerSnapshot` | Alias or subset view of **`BraidSchedulerDiagnostics`** for documentation/tooling clarity (**may be redundant** with existing type). |

### 4.3 xUnit-oriented (**optional package `Braid.Xunit` or similar**)

| Candidate | Idea |
|-----------|------|
| `BraidXunit.WriteFailure(ITestOutputHelper output, BraidRunException ex)` | Writes a single bounded, clearly delimited block (e.g. markdown-friendly or plain `---` sections). |
| `BraidXunit.AssertThrowsRunAsync(...)` | Thin wrapper around **`RunAsync`** + **`ThrowsAsync<BraidRunException>`** that automatically appends **`ToReportString()`** on failure (**easy to over-engineer** — keep minimal if shipped). |

**Package trade-off:** keeping **xUnit** out of **`braid`** core preserves fewer dependencies for library consumers; an optional package aligns with [roadmap](../roadmap.md) v0.4.0 notes.

## 5. Proposed implementation split — 5 small PRs

| PR | Focus | Outcome |
|----|--------|---------|
| **PR1 — Report formatting refactor** | Extract internal/shared logic between **`BraidRunException.ToString()`** and any new formatter; add golden **`ToString`** tests if layout changes; **optional** `TryGetReplayText`-style API **if** approved. | Single place for replay-text + diagnostics sections; reduced duplication; **no scheduler changes**. |
| **PR2 — Structured report builder** | Add **`BraidFailureReport`** (or static factory `BraidFailureReport.From(BraidRunException)`) **if** approved — read-only, testable without string parsing. | Assertions on properties instead of **`Contains`** on monolithic strings where desired. |
| **PR3 — Trace contract (minimal)** | Either document trace line grammar only **or** add optional **`BraidTraceEntry`** / parsed view behind a non-breaking API — **decision in PR**. | Clearer contract for tools; **scheduling unchanged**. |
| **PR4 — `Braid.Xunit` (optional)** | New package: **`ITestOutputHelper`** helper + README snippet; reference **`Microsoft.NET.Test.Sdk`** / **xunit.v3** as appropriate; **no** forced dependency from core. | Better defaults for test authors. |
| **PR5 — Docs + example** | **`docs/`** workflow: flaky random failure → capture seed/trace → add probes → **`Parse`** / typed replay; link from README; optional small example test project pattern. | Socializes v0.4.0 without changing semantics. |

If fewer PRs are preferred, **merge PR2 into PR1** or **defer PR3** until demand exists — but keep **scheduler PRs forbidden** in this epic unless a regression requires it.

## 6. Exit criteria

- **Semantics:** Replay, random, Arrive/Hit/Release, timeout, and cancellation behavior **unchanged** unless covered by a dedicated bugfix release with explicit changelog entries.  
- **Discoverability:** README or **`docs/`** describes how to obtain replay text and diagnostics **without** relying solely on reading **`ToString()`** source.  
- **xUnit:** Documented one-liner (or package helper) to emit braid diagnostics on failure.  
- **Honesty:** Random-only runs still **do not** synthesize a full replay schedule; docs and APIs remain explicit.  
- **Non-goals:** Still no TaskScheduler replacement, await interception, binary rewriting, or exhaustive model checking as part of this workstream.
